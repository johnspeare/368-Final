Selected samples of content that I delivered while working at Microsoft

Suite

Owning a suite of content was the rewarding and challenging part of being a Senior Techncial Writer. Generally, a suite of content comes to life by its relation to a technical feature set, but ultimately, a set of features maps to a customer scenario.

Driving a suite of content to publication with high quality requires unrelenting customer empathy and a thorough understanding of the problem your audience is attempting to solve and the context in which they are attempting to resolve it.

  • Writer: Exchange/Office 365 directory integration scenarios. Exchange Hybrid Deployment and Migration with Office 365, the "Cloud only" scenario in the Exchange Deployment Assistant. Configure directory synchronization.
  • Writer: Exchange 2010 Certificates
  • Manager, Writer, Technical Reviewer: Mobile Information Server
Technical Reference

Technical reference content is used by system administrators for specific configuration tasks. Generally, this documenation is very system-centered and light on context or scenario description.

For example, Exchange Server shell content is organized according to system components. Each component has a set of commands ("cmdlets"), which in turn take a predefined set of parameters. Cmdlet help ships on the web (see first link below), but it also ships in the Exchange management components where the administrator can access it from within the command shell.

  • Writer: Exchange 2010 Anti-Spam Cmdlets
  • Writer, SME Editor: Working with Active Directory Permissions in Exchange Server
Conceptual

Conceptual help content is developed according to a specific audience scenario. Generally, this type of content does not contain procedures; it may introduce a feature set, describe how a given feature fits into a broader solution, or explain foundational technology concepts.

  • Writer: Spam Filtering and Message Hygiene
  • Writer: Exchange Security Documentation: What’s New and Who’s Who
Procedural

Procedural content explains how the reader can solve a specific task. This kind of content is what most people assume techincal communication is all about. However, the longer I spent writing content at Microsoft, the less procedural content I delivered.

  • Writer: Create Accepted Domains – Live@edu
  • Writer: Install and Upgrade the Microsoft Online Services Directory Synchronization Tool
Whitepaper

The whitepaper: the traditional, standalone content aimed at a specfic scenario, problem, or feature set. True to their roots, whitepapers often ship as downloadable documents that are optimized for printing, but the two samples here live online only.

  • Writer: Exchange 2010 Security Guide
  • Writer: Domain Security in Exchange 2007
Print

When it comes to software documentation, print is now mostly dead. But when I started at Microsoft in 1998, big books still "shipped in the box." I worked on a few of these, and a couple Resource Kits. I also contributed to content that went into some magazines.

  • Writer:   Multiple chapters and sections, Exchange 2000 Server: Planning and Installation Guide, Microsoft Corporation (“in box” manual)
  • Writer, Technical Reviewer, Editor:   Multiple chapters and sections, Microsoft Exchange 2000 Server Resource Kit, Microsoft Press
  • Technical Reviewer:   Multiple chapters and sections, Introducing Microsoft Exchange 2000 Server, by JoAnne Woodcock, Microsoft Press
  • Technical Reviewer:   TechNet Magazine, May 2007, Microsoft
